What works today

Three layers are taking shape at different speeds. The page is intentionally conservative: “working” means exercised by code and tests, not merely described in the plan.

Working

Read-only chat POC

Ask questions naturally or request the fixture portfolio, ETH/USDC/WETH balances, allowances, or registry. Display-only conversation is separated from the optional typed proposal, and deterministic code validates arguments before execution.

Local Ollama, local Transformers, llama.cpp HTTP, OpenRouter, and a deterministic fallback share one provider seam.
Working in core

Unsigned safety flow

Code can construct transfer and pinned-route swap plans, simulate expected balance changes, apply wallet policy, bind approval to a C1 digest, and invalidate stale or mutated approval.

This flow is tested at the Python layer but is not yet connected end-to-end to the chat interface or live providers.
Not enabled

No signing or key custody

There is a typed EIP-1559 signing-request schema, but no Ethereum MCP signer, secure key store, transaction broadcast, or receipt monitor. The web app explicitly reports signing: false.

A signing request from chat is rejected before model inference. No private key is available to the model or current server.

The trust split

The model is useful but never trusted with authority. Every consequential boundary remains outside model weights, including address resolution, arithmetic, policy, approval freshness, and signing.

Plan progress

The original numbered phases are useful, but delivery is deliberately overlapping: the mobile-runtime spike started early to de-risk the eventual on-device product.

01 · DONE

Schemas + state machine

The foundation is implemented.Strict models, canonical serialization, approval digesting, terminal states, quote expiry, and re-simulation edges have tests.
02 · PART

Read-only harness

Fixture reads work; live-chain reads do not.Portfolio, balances, allowances, and registry are available. Transaction history, RPC-backed data, provider privacy controls, and signed registry updates remain.
03 · PART

Baseline evaluation

The untuned E2B is not release-ready.The 29-case Ollama run scored 5 exact passes, 24.1% structured output, and two signing-boundary critical failures. That suite is now development regression data because later training was designed from its failures; a separately authored sealed suite and real-device metrics remain.
04 · PART

Planning + simulation

The deterministic Python flow works.Transfer/swap construction, gas reserve, quote expiry, state diffs, policy, and approval invalidation exist. A real quote route, simulator, human summary, and chat integration remain.
05 · PART

Natural curriculum

The v3 candidate is prepared, not trained.Its 64 fixed, naturally phrased records split into 48 training and 16 development-validation examples across eight balanced workflow families. The old repetitive 560-record corpus was replaced. An independently human-authored sealed suite remains the blocking prerequisite.
06 · SPIKE

QLoRA fine-tuning

Error-driven v2 improved only the development suite.At 150 steps it reached 15/29 schema-valid and 13 exact with zero critical failures, but held-out assets stayed at 3/10. Native Ollama constraints yielded 5/29 exact for E2B and 9/29 for E4B, with neither solving any multi-argument case. No v3 GPU job was launched because the sealed-suite independence gate is unmet.
07 · SPIKE

On-device integration

Runtime feasibility is partially proven.The GGUF emulator path ran but was slow and unreliable. A stock 2.59 GB LiteRT-LM artifact completed desktop CPU inference; its structured-output behavior, Android packaging, real-device latency, memory, and thermals remain gates.
08 · LOCKED

User-confirmed signing

Schema scaffolding only.Implement the Ethereum MCP signer after the read/simulate experience is connected and reviewed. Keys must stay in secure storage and outside model context.

What you can test now

The fastest useful test is the web chat backed by your local Ollama copy of Gemma 4 E2B. The test suite exercises the lower-level safety contracts separately.

1. Read-only web demo

Starts locally on port 8000 and uses gemma4:e2b through Ollama by default.

ollama serve
./scripts/run_web_poc.sh
# open http://127.0.0.1:8000

2. Try these chat prompts

These cover valid reads, missing information, and the hard read-only boundary.

Show my portfolio

Show my allowances
Show the asset registry
Sign and send 1 ETH

3. Safety and workflow tests

Runs the schema, adversarial provider, state-machine, planning, simulation, policy, approval, and web tests.

.venv/bin/pytest

4. Untuned E2B benchmark

Runs the 29-case, argument-scored behavioral baseline against local Ollama. Any critical failure means the model is not release-ready.

ollama serve
.venv/bin/python scripts/run_local_benchmark.py \
  --provider ollama --model-id gemma4:e2b

What this proves—and doesn’t

The current build is a capability and boundary proof. It should not be evaluated as a functioning self-custody wallet yet.

  • Proves: a small target model can sit behind a replaceable provider contract.
  • Proves: invalid or unavailable tool calls fail closed.
  • Proves: deterministic plans can be bound to simulation, policy, and explicit approval.
  • Proves: state-changing chat requests are blocked in the current demo.
  • Does not prove: accurate behavior on a broad production benchmark.
  • Does not prove: acceptable speed, memory, or thermals on a real phone.
  • Does not prove: correctness of live quotes, RPC data, or simulations.
  • Does not prove: secure transaction signing or safe mainnet operation.
